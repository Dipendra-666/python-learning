what is the difference between container and vm?
=>Containers and virtual machines both provide isolated environments to run applications, but they differ in how they achieve that isolation.
 A virtual machine runs on top of a hypervisor and includes a full guest operating system, which makes it heavier and slower to start, but
 provides strong isolation. On the other hand, containers run on top of the host operating system and share its kernel, using tools like
 Docker. Because they don’t require a full OS, containers are lightweight, start very quickly, and are more efficient in terms of
 resource usage. However, they offer slightly less isolation compared to VMs. In practice, VMs are preferred when strong isolation
 or different operating systems are needed, while containers are widely used for microservices,
 CI/CD pipelines, and scalable applications due to their speed and efficiency.

what are the lifecycle of docker?
=>Docker lifecycle refers to the stages an application goes through from building the image to running and managing containers using a tool like Docker.
It starts with writing a Dockerfile, where we define the application environment—like base image, dependencies, and commands. Then we build the Docker image from that Dockerfile. This image is a read-only template that contains everything needed to run the application.
Once the image is built, we can optionally push it to a registry like Docker Hub so it can be shared or deployed in different environments. After that, we create and run containers from the image. A container is a running instance of that image.
During runtime, we can manage the container—start, stop, restart, or monitor it depending on requirements. If needed, we can also update the application by modifying the Dockerfile, rebuilding the image, and redeploying new containers.
Finally, when the container is no longer needed, we stop and remove it, and optionally clean up unused images to free resources.

what are the differnt docker components?
=> Docker components are the core parts that make up how Docker works, mainly including the client, daemon, images, containers, and registry.

First, the Docker Client is what we use to interact with Docker using commands like docker build or docker run. It sends these requests to the 
Docker Daemon (the Docker engine), which is the backend service responsible for actually building images, running containers, and managing everything.

Then we have Docker Images, which are read-only templates that contain the application and its dependencies. From these images, we create Containers,
 which are the running instances of those images.

Docker also uses a Registry, like Docker Hub, to store and share images. We can push images to the registry and pull them whenever needed.

Additionally, there are supporting components like volumes for persistent storage and networks for communication between containers.


what is the difference between docker add and docker copy?
=>Both COPY and ADD are instructions used in a Dockerfile to move files into an image when building with Docker, but they have some key differences.

COPY is simple and predictable—it only copies files or directories from the local system into the container image.

ADD, on the other hand, has additional features. It can not only copy files, but also automatically extract compressed files like .tar, and it can fetch files from a URL.

Because ADD has these extra behaviors, it can sometimes be less predictable. So in best practices, we usually prefer COPY unless we specifically need those extra features like auto-extraction.



what is the difference between ENTRYPOINT and CMD ?
=>Both CMD and ENTRYPOINT in a Dockerfile (used with Docker) define what command runs when a container starts, but they behave differently.

CMD is used to provide default arguments or a default command, and it can be easily overridden when running the container using docker run.

ENTRYPOINT, on the other hand, defines the main command that will always run. It’s not overridden by default—instead, any arguments passed in docker run are appended to it.

So the key difference is:

CMD → flexible, can be overridden
ENTRYPOINT → fixed, acts as the main executable

In practice, we often use them together—ENTRYPOINT for the main command, and CMD for default arguments
